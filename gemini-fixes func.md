# ТЗ на доработку программы учета автозапчастей

## Контекст для LLM
Программа представляет собой локальное приложение для учета заказов автозапчастей. Пользователь (сотрудник магазина) дал обратную связь по текущей версии. Необходимо внести исправления в интерфейс, логику поиска, экспорт данных и добавить финансовый блок для частичной оплаты, не сломав существующий функционал (включая работу с Атрибутами клиентов).

## Задача 1: Исправление формата экспорта базы (CSV для Excel)
**Проблема:** При открытии скачанного CSV файла в Microsoft Excel все данные слипаются в одну колонку.
**Решение:**
1. При генерации CSV файла убедиться, что в самом начале строки добавляется символ BOM (Byte Order Mark) `\uFEFF`, чтобы Excel корректно читал кириллицу (UTF-8).
2. Использовать в качестве разделителя столбцов строго **точку с запятой (`;`)**, а не обычную запятую (`,`).
3. Обернуть значения текстовых полей в кавычки, если внутри текста потенциально может встретиться точка с запятой.

## Задача 2: Принудительный запуск в Google Chrome (Лаунчер)
**Проблема:** Программа по умолчанию открывается в стоковом браузере Windows (Edge), а пользователю нужен Chrome.
**Решение:**
Написать скрипт для запуска (например, `start.bat` или обновить логику Python-лаунчера). Скрипт должен пытаться найти путь к Chrome в Windows и запускать интерфейс в нем. Если Chrome не найден, использовать браузер по умолчанию.

## Задача 3: Добавление поиска в разделы "Склад" и "Поставщики"
**Проблема:** В разделах "Склад" и "Поставщики" отсутствует строка поиска, при большом количестве позиций работать невозможно.
**Решение:**
1. Добавить UI-элемент (строку поиска) в верхнюю часть разделов "Склад" и "Поставщики" по аналогии с разделом заказов.
2. Написать функцию фильтрации таблиц/списков "на лету" при вводе текста.

## Задача 4: Поле "Артикул" и глобальный поиск
**Проблема:** В карточке товара на складе есть только "Наименование", не хватает отдельного поля под артикул.
**Решение:**
1. В форму добавления/редактирования товара на Складе добавить новое обязательное текстовое поле `Артикул` (под полем `Наименование`).
2. Обновить структуру базы данных склада, чтобы артикул сохранялся как отдельное свойство.
3. Настроить поиск по Складу (из Задачи 3) так, чтобы он искал совпадения и по полю `Артикул`.

## Задача 5: Логика частичной оплаты и расчет долга в Заказах
**Проблема:** В форме редактирования заказа (когда клиент вносит предоплату или оплачивает заказ частично) не хватает полей для фиксации этой суммы и расчета остатка долга.
**Решение:**
1. В форму добавления/редактирования заказа добавить поле ввода `Внесенная предоплата` (числовое).
2. Рядом добавить поле (или текстовый блок) `Остаток долга` (Только для чтения / Read-only).
3. Написать реактивную логику (JS/Frontend): при вводе суммы в поле "Внесенная предоплата" или изменении итоговой "Цены продажи", программа должна моментально автоматически пересчитывать и отображать значение в поле "Остаток долга".
   *Формула: `Остаток долга = Цена продажи - Внесенная предоплата`.*
4. Убедиться, что эти новые финансовые данные (предоплата и долг) корректно сохраняются в базу данных заказа и выгружаются при экспорте в CSV (Задача 1).
